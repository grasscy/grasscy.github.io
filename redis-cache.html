读写策略

cache aside pattern（旁路缓存模式）：读较多

写

  • 更新 db
  • 删除缓存
先删再写？
  • 不行，会产生不一致
    • A 删数据
    • B 从 db 读数据——脏数据存入了 cache
    • A 更新数据
先写再删一定没问题？
  • 否，几率很小，因为缓存写入速度比 db 写入快得多
    • A 从 db 读数据
    • B 更新 db 并删 cache
    • A 写 cache

读

  • 读 cache，读到则直接返回
  • cache 无，之读 db
  • 把数据存入 cache

问题

  • 首次一定不在 cache
    • 热点数据提前放入 cache
  • 写操作频繁时，cache 被频繁更新，影响 cache 命中率
    • 强一致场景：加锁
    • 允许不一致：给 cache 加个较短的过期时间

read/write through pattern(读写穿透模式)

写

  • cache 中不存在，直接更新 db
  • cache 中存在，先更新 cache，然后 cache 服务自己更新 db

读

  • cache 存在，读到直接返回
  • cache 不存在，cache 服务自己从 db 加载，写入后返回

问题

  • 首次一定不在 cache
    • 热点数据提前放入 cache

write behind pattern（异步缓存写入）

  • 只更新缓存，异步批量的方式更新 db，如 redis 异步写入磁盘、innodb buffer pool
  • 有一致性问题
  • 写入性能高，适用数据经常变化有对一致性要求没那么高的场景，如浏览量、点赞量

问题

  • 缓存雪崩
    • 缓存同一时间大面积的失效，请求打到数据库上
    • 解决办法
      • 设置不同的缓存时间
      • 更新数据使用互斥锁
      • 通过双缓存
  • 缓存击穿
    • 高并发访问时缓存过期，请求打到数据库上
    • 解决办法
      • 使用互斥锁，只让一个请求 load db，成功后写缓存；其余尝试重新获取缓存中数据
  • 缓存穿透
    • 请求缓存中不存在的数据，请求打到数据库上
    • 解决办法
      • 互斥锁
      • 无论是否取到结果都存缓存，过期时间较短（如 5min）
      • 采取机制（如布隆过滤器、bitmap）拦截非法 key
        • 布隆过滤器：可以判断一定不存在，但无法 100%判断一定存在
  • 双写一致性
    • 要求强一致性，建议不要用缓存
    • 数据库中或缓存数据的删除和写入过程中，如果有失败的情况，会导致数据的不一致
    • 解决办法
      • 双删延时+缓存超时：先删缓存，再更新数据库，最后再隔一定的时间删除缓存
        • 确保读请求结束，写请求可以删除读请求造成的缓存脏数据。
        • 当然这种策略还要考虑 redis 和数据库主从同步的耗时。
        • 最后的的写数据的休眠时间：则在读数据业务逻辑的耗时基础上，加⼏百 ms 即可。⽐如：休眠 1 秒。
      • 异步更新缓存：订阅 binlog 同步机制（canal）
        • binlog 增量（update,insert,delete)订阅消费
        • 消息队列
        • 增量数据更新到 redis
