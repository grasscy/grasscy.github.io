HashMap——兼以Hashtable

概述

HashMap和Hashtable同作为常见的Map实现，两者的区别在于HashMap支持key=null、value=null而Hashtable不支持；HashMap不支持同步而Hashtable采用synchronized修饰方法。除此之外，两者提供的功能几乎相同。两者都采取array+list的方式存储键值对Entry，array[i]事实上存储了一个list的头结点。但是，相对于Hashtable，HashMap会在array.length(称为capacity)和list.size()都达到一定程度之后，将list转换成红黑树；而红黑树的结点低于一定数量的时候，也会重新转换成list。

本文着重HashMap与Hashtable，并不讨论红黑树相关。如有需要，请移步红黑树相关文章。

为了方便叙述，本文中table指存储Entry的数组结构(Entry[] table)；bin指table[i]和其list或红黑树；capacity指table.length，也就是bin的数量；threshold指达到再散列的门槛size(Entry总数)。

HashMap的遍历实现与LinkedHashMap十分相似，区别在于LinkedHashMap的nextNode根据e.after确定，而HashMap由于不需要保证遍历顺序，直接使用nextNode=e.next。具体请参考LinkedHashMap篇，这里也不在叙述。

概述

Hashtable属于早期java的集合，后来集合框架发生了变化，因此Hashtable通常被新的集合框架下的HashMap所取代。尽管相对于HashMap，Hashtable采取了同步。但是在有同步需求的情况下，也不推荐用Hashtable，而是使用ConcurrentHashMap。

Hashtable中，使用Entry<?,?>[] table来存储键值对，而每个Entry也是一个单向链表的结点。可以说，table[index]实际上指向了一个Entry的链表的头结点。新插入的Entry，则占据table[index]，成为Entry链表的头结点。

AbstractList/ArrayList/LinkedList

导航

  • AbstractList
  • ArrayList
  • LinkedList

概述

ArrayList和LinkedList是常用的集合，都是AbstractList子类。但由于ArrayList以数组实现，而LinkedList以双向链表实现，所以两者的对于诸如get(index)之类的带下标操作有极大不同：ArrayList可以使用index直接访问（implements RandomAccess），而LinkedList只能通过顺序访问（extends AbstractSequentialList）。

AbstractList

  • modCount
    AbstractList中设置modcount字段，让Iterator通过expectedModCount == modCount来判断是否有另外的线程更改了list的结构（structurally modified：list的size发生了变化）。

  • subList(int fromIndex, int toIndex)

所以，class SubList extends AbstractList 实现了其中的主要方法。SubList存储了原list引用l，其中方法对l进行操作。所以对sublist的操作，将会同时影响原list。也因此，sublist用this.modCount == l.modCount来checkForComodification()

ArrayList

某种程度上，List可以看做自动扩展容量的数组。以数组实现的ArrayList，不可忽视的便是容量的扩展操作。ArrayList中，当需要的最小容量不足时，将会先扩展至原容量的1.5倍。仍然不足，才会扩展至需求的最小容量。

LinkedList

采用双向链表实现，在除了能实现List接口，也能很方便的实现Deque接口。对LinkedList的各种操作，主要在于节点的link和unlink，而其中占多数的，是对头结点和为节点的操作，如List的add，以及Deque的各种方法。

AbstractStringBuilder/StringBuilder/StringBuffer

概述

抽象类AbstractStringBuilder实现了Appendable, CharSequence接口，StringBuilder和StringBuffer继承AbstractStringBuilder类并额外实现了java.io.Serializable。
AbstractStringBuilder类中实现了大量方法，并被子类所继承。其中子类StringBuffer以synchronized保证了所有方法的同步，并利用char[] toStringCache来提高toString的性能。

为什么StringBuilder不采用toStringCache呢？参考stackoverflow,大意是StringBuiler不提供线程同步，而在保持线程不同步的性能优势下，提供缓存极难实现。并且，这种缓存优化现实中使用很少。

jdk-String

Posted on 2018-10-18 | In java , java源码

String

前言

要理解String,需要对字符编码有一定了解。最早的计算机采用8字节作为一个byte，由此诞生了采用一个byte的表示字符的编码：ASCII编码。
在世界范围内，1byte的表示范围显然是远远不够的，为了统一所有文字的编码，Unicode应运而生。
最初，Unicode设计成16字节形式，可以表示65536个字符。而后随着内容的增加，这65536个字符合集便被称为基本多文种平面（Basic Multilingual Plane,BMP），并添加了要求以2-byte实现的辅助平面（Supplementary Planes）。
按照Unicode最长的标准来实现，无疑会造成巨大的空间浪费。于是实现了Unicode标准的变长编码出现了，如utf-8/utf-16等。
utf-16中，以2-byte表示BMP，4-byte表示SP。而java中，.class采用utf-8,JVM采用utf-16。
code point是unicode标准下的编码，code unit是编码实现中code point的存储单元，在utf-16中，意味着2-byte。这在java中，被命名为char。

如length()返回数组长度，也就是code unit数量。

char的包装类Character实现了遵循utf-16的各种方法，而在String类中，以char[] value表示值，调用Character实现方法。如进行code point判断、大小写转换等。

  • contentEquals(CharSequence cs)
  • indexOf(String str, int fromIndex)
  • split(String regex, int limit)

notify()/wait()

在JAVA中，是没有类似于PV操作、进程互斥等相关的方法的。JAVA的进程同步是通过synchronized()来实现的，需要说明的是，JAVA的synchronized()方法类似于操作系统概念中的互斥内存块，在JAVA中的Object类型中，都是带有一个内存锁的，在有线程获取该内存锁后，其它线程无法访问该内存，从而实现JAVA中简单的同步、互斥操作。明白这个原理，就能理解为什么synchronized(this)与synchronized(static XXX)的区别了，synchronized就是针对内存区块申请内存锁，this关键字代表类的一个对象，所以其内存锁是针对相同对象的互斥操作，而static成员属于类专有，其内存空间为该类所有成员共有，这就导致synchronized()对static成员加锁，相当于对类加锁，也就是在该类的所有成员间实现互斥，在同一时间只有一个线程可访问该类的实例。如果只是简单的想要实现在JAVA中的线程互斥，明白这些基本就已经够了。但如果需要在线程间相互唤醒的话就需要借助Object.wait(), Object.nofity()了。

Obj.wait()，与Obj.notify()必须要与synchronized(Obj)一起使用，也就是wait,与notify是针对已经获取了Obj锁进行操作，从语法角度来说就是Obj.wait(),Obj.notify必须在synchronized(Obj){…}语句块内。从功能上来说wait就是说线程在获取对象锁后，主动释放对象锁，同时本线程休眠。直到有其它线程调用对象的notify()唤醒该线程，才能继续获取对象锁，并继续执行。相应的notify()就是对对象锁的唤醒操作。但有一点需要注意的是notify()调用后，并不是马上就释放对象锁的，而是在相应的synchronized(){}语句块执行结束，自动释放锁后，JVM会在wait()对象锁的线程中随机选取一线程，赋予其对象锁，唤醒线程，继续执行。这样就提供了在线程间同步、唤醒的操作。Thread.sleep()与Object.wait()二者都可以暂停当前线程，释放CPU控制权，主要的区别在于Object.wait()在释放CPU同时，释放了对象锁的控制。

finalize()

  • JVM判断该对象不可到达且未死亡时调用
  • 调用完成后，JVM再次扫描，该对象仍不可到达且未死亡才可能被回收
  • 抛出异常，该异常将会无视，并且finalize（）终止
